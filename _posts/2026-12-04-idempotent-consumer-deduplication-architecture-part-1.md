Most message systems are honest if you read the fine print:
delivery may be at least once, retries happen, and duplicates are part of normal operation.

The real question is not whether duplicates exist.
It is whether your consumer can absorb them without corrupting business state.

That is why idempotent consumer design matters.
It turns duplicate delivery from a hidden bug source into an explicit boundary decision.

## Quick Summary

| Question | Good baseline answer |
| --- | --- |
| What is the unit of idempotency? | one business operation, identified by a stable key |
| Where is dedup recorded? | in the same durability boundary as the side effect when possible |
| What usually goes wrong? | dedup store and side effect commit in different places |
| What does idempotency not mean? | "the broker will only deliver once" |

The invariant is simple:
reprocessing the same logical message must not apply the business effect twice.

## Exactly-Once Is Usually the Wrong Mental Model

Teams say "we need exactly once" when they usually mean one of three things:

- do not charge twice
- do not create duplicate orders
- do not increment counters twice

Those are business invariants.
They are not the same as transport-level exactly-once delivery.

A healthier baseline is:

- assume duplicates can happen
- define a stable idempotency key
- make duplicate handling a normal path, not an exception path

That mindset produces better systems than betting the design on a marketing phrase.

## Start With the Idempotency Key

Everything gets easier once the key is explicit.

Good keys are:

- stable across retries
- tied to the business operation
- available to the consumer at processing time

Examples:

- payment authorization request ID
- order placement command ID
- upstream event ID plus producer identity

Bad keys are:

- local database auto-increment values generated by the consumer
- timestamps with insufficient uniqueness guarantees
- payload hashes when field ordering or metadata drift can change them

If the key is unstable, the rest of the dedup design is built on sand.

## The Best Storage Pattern

The safest common pattern is:
persist "I processed this idempotency key" in the same transactional boundary as the business state mutation.

That often looks like:

1. begin transaction
2. insert idempotency key into a table with a uniqueness constraint
3. apply business mutation
4. commit

If step 2 fails because the key already exists, treat the message as already processed.

```java
public final class PaymentSettledConsumer {
    public void handle(PaymentSettledEvent event) {
        transactionTemplate.executeWithoutResult(status -> {
            boolean inserted = dedupRepository.tryInsert(event.eventId());
            if (!inserted) {
                return;
            }

            ledgerRepository.markSettled(event.paymentId(), event.settledAt());
        });
    }
}
```

The important design property is not the Java syntax.
It is that dedup and business effect succeed or fail together.

## Where Dedup Breaks Down

The dangerous version is:

1. check Redis for event key
2. write business row to database
3. write dedup key to Redis

That looks fine until the database write succeeds and the dedup write fails.
Now the retry sees no dedup key and applies the side effect again.

This is why teams should treat "where is the dedup record stored?" as a correctness question, not an implementation detail.

## Good Dedup Strategies by Scenario

| Scenario | Strong approach | Why |
| --- | --- | --- |
| database-backed business mutation | unique key in same database transaction | strongest practical baseline |
| append-only ledger or immutable events | natural business key or sequence check | avoids separate dedup path |
| high-throughput short-lived dedup window | bounded dedup store plus TTL | cheaper when permanent storage is unnecessary |
| external side effect like email | idempotency key propagated to downstream provider | consumer alone cannot guarantee safety |

The right design depends on whether the side effect is local, transactional, and replay-safe.

## Idempotent Consumer Is Not Just a Broker Concern

Even if the broker provides ordering or transactional semantics, consumers still need a business-level duplicate strategy because:

- replay jobs exist
- manual recovery exists
- dead-letter redrives exist
- downstream systems may retry independently

A system that is safe only when the broker behaves perfectly is not production-safe.

## Operational Failure Modes to Design For

### Dedup retention is too short

The replay arrives after the dedup key expired.
Now yesterday's message is treated as new.

### Consumer version change alters the dedup rule

One deploy uses `eventId`, the next uses `paymentId`.
Now duplicates sneak past because the identity contract drifted.

### Side effects happen outside the dedup boundary

The database update is idempotent, but the notification call is not.
The system is only partially safe.

### Ordering is mistaken for idempotency

Ordered partitions reduce some complexity.
They do not remove the need to survive retries and redelivery.

## Observability That Actually Helps

Expose:

- duplicate detection count
- dedup insert conflicts or uniqueness violations
- dedup store lookup latency
- expired-key replay count
- downstream side-effect retry count
- ratio of duplicates by producer, topic, or partition

That turns the design into something operators can reason about.
If duplicate handling is invisible, the first serious replay becomes archaeology.

## A Practical Decision Rule

Use idempotent-consumer design when the same logical message may be retried or replayed and the business effect is not naturally safe to apply twice.

Prefer the simplest boundary that makes the invariant true:

- same transaction when possible
- natural key when available
- downstream propagated idempotency when the side effect is external

Do not scatter dedup logic across multiple stores just because each piece seems convenient locally.

## Part 1 Checklist

- a stable idempotency key is defined
- dedup record and business mutation share a durability boundary where possible
- duplicate processing is a normal code path
- retention window matches replay reality
- downstream side effects have their own idempotency strategy
- dashboards expose duplicate rate and dedup health

## Key Takeaways

- Idempotent consumer design is about business safety, not transport slogans.
- The idempotency key is the first design decision, not an afterthought.
- Dedup is safest when recorded in the same transaction as the business effect.
- If retries are normal but duplicate handling is hidden, the architecture is not finished.
